## A23 — TalkBack / VoiceOver

### Code-level baseline

- 32 Compose files include `contentDescription` or `Modifier.semantics`.
- 28 SwiftUI files include `accessibilityLabel` / `accessibilityIdentifier` / `accessibilityHint`.
- Total Compose @Composable screen files: 27. Total SwiftUI screen files: 25. Coverage ratio > 100% because many files include multiple annotated subcomponents.

### Pre-alpha drill (operator action)

Required walk-through (cannot be automated):

1. Enable VoiceOver on iOS (Settings → Accessibility → VoiceOver → ON).
2. Enable TalkBack on Android (Settings → Accessibility → TalkBack → ON).
3. Walk through every primary surface in this order:
   - Onboarding (3 pages + Skip / Get Started)
   - Sign In / Sign Up / Forgot Password
   - Pattern Library + Patterns CTA
   - Pattern Detail
   - Chart Viewer
   - Project List + Project Detail + Progress
   - Discovery (Browse Patterns) + search + filter
   - Suggestion list + Suggestion detail + Apply / Close
   - Variations list + Versions list + Diff view
   - Settings (every row in every section)
   - Pro paywall
4. For each surface, log:
   - Unlabeled IconButton / IconToggle ("button button button" without role).
   - Decorative images that should be `contentDescription = null` (Compose) or `.accessibilityHidden(true)` (SwiftUI).
   - Composite components that announce field-by-field instead of as a logical unit.
5. File the result as `audits/a11y-walkthrough-<DATE>.md`.

**Expected effort**: 2-4 hours single pass for both platforms.

**Verdict for pre-alpha launch**: Walk-through is **not a launch blocker** at closed-alpha scale (5–10 testers, most of whom are not screen-reader-primary users). Defer walk-through until alpha testers are recruited; gate post-alpha widening on completion. The 32+28 code-level annotation baseline is good enough for stock components to announce sensibly.

## A24 — Dynamic Type / font scaling

### Code-level baseline

Zero explicit references to font-scaling APIs:
- iOS: no `@ScaledMetric`, no `UIFontMetrics`, no `Environment(\.sizeCategory)`.
- Android / Compose: no `LocalDensity.current.fontScale`, no `TextUnit.Em`-based responsive sizing.

This is *not* a bug — Material 3 and SwiftUI both auto-scale `Text` / `TextField` per the system font scale by default. The risk is in **overrides**: if any screen sets a fixed pixel font size or a fixed-height row that text outgrows, scaling will break.

Skeinly's strings use Material 3 typography tokens (`MaterialTheme.typography.titleLarge`, `bodyMedium`, etc.) and SwiftUI dynamic-type defaults (`.font(.title2)`, `.body`, `.footnote`). Both are scale-respecting by default.

### Pre-alpha drill

1. Set iOS to maximum text size (Settings → Accessibility → Display & Text Size → Larger Text → drag slider to max).
2. Set Android to "Largest" font size (Settings → Display → Font size → Largest).
3. Launch Skeinly on each platform; walk through the same 11 surfaces from A23.
4. Log any:
   - Text truncation or clipping.
   - Overlapping elements where larger text pushes one element into another.
   - Buttons with fixed heights that text outgrows (button label gets clipped).
5. Fix on a per-screen basis: replace fixed dp/pt with relative sizing or `Modifier.heightIn(min = …)` to grow with content.

**Verdict for pre-alpha launch**: Smoke test before invite. Likely zero blockers since no overrides exist; the smoke test catches the few places that might break.

## A25 — Reduce Motion

### Code-level baseline

**Zero references** to `UIAccessibility.isReduceMotionEnabled` (iOS), `AccessibilityManager.isAnimationsEnabled` (Android), or any Compose-side helper.

### Animation surfaces inventory

- **`OnboardingScreen.kt`** (Compose): `AnimatedVisibility` / page slide transitions when advancing through the 3 onboarding pages. The only commonMain animation that's currently outside Material 3 defaults.
- **iOS SwiftUI** (4 files using `withAnimation` / `TransitionSlide`): the in-app banner slides + the Settings → notifications row state transition + the paywall sheet present / dismiss animation.
- **Material 3 default transitions** (modal sheets, navigation, list animations): respect the system Reduce Motion setting automatically on iOS (`UIAccessibility.isReduceMotionEnabled` is consulted by UIKit-bridged Composables). On Android, Compose Material 3 respects `Settings.Global.ANIMATOR_DURATION_SCALE` automatically. **Stock Material 3 transitions = automatically gated; no code needed.**

### Wave D code action

Wave D Iteration 2 will introduce a small `expect/actual` helper:

```kotlin
// commonMain
expect fun isReduceMotionEnabled(): Boolean

// iosMain
actual fun isReduceMotionEnabled(): Boolean =
    UIAccessibilityIsReduceMotionEnabled()

// androidMain
actual fun isReduceMotionEnabled(): Boolean =
    // AccessibilityManager has the isAnimationsEnabled getter on API 33+;
    // earlier targets fall back to Settings.Global.ANIMATOR_DURATION_SCALE.
    …
```

And gate the OnboardingScreen page-slide animation behind it:

```kotlin
val transitionSpec = if (isReduceMotionEnabled()) snap() else tween(durationMillis = 300)
```

iOS-side: the 4 SwiftUI files using `withAnimation` will be inspected; each animation either (a) already respects Reduce Motion via SwiftUI default behavior, or (b) is wrapped in `withAnimation(.linear(duration: 0))` when `UIAccessibility.isReduceMotionEnabled == true`.

**Wave D scope**: ship the expect/actual helper + apply to the single Compose commonMain surface (`OnboardingScreen`). iOS sweep batched into a separate Wave (different blast radius; iOS sweep is 4 files vs commonMain 1).

### Closure (2026-05-16 — iOS SwiftUI sweep)

Compose half closed in Wave D (2026-05-12): `ReduceMotionDetector`
`expect/actual` + `OnboardingScreen.kt` page-slide gate
(`if (reduceMotion) snap()`). The iOS SwiftUI sweep closes the
remaining half.

The 2026-05-12 *Animation surfaces inventory* line above ("in-app
banner slides + Settings → notifications row state transition + paywall
sheet present/dismiss") was a pre-sweep estimate; the codebase evolved
across later phases. The actual sweep enumerated every SwiftUI
`withAnimation` / animated slide `.transition` and resolved exactly the
4 in-scope files:

| File | Motion | Resolution |
|---|---|---|
| `Screens/SuggestionDetailScreen.swift` | `withAnimation` ×4 — close/merge success-toast show+hide | `withMotion(reduceMotion)` — instant toggle under RM; `.transition(.opacity)` becomes inert (no transaction) |
| `Screens/ChartConflictResolutionScreen.swift` | `withAnimation` ×2 — merge-applied toast show+hide | same `withMotion(reduceMotion)` treatment |
| `Components/ChartImageViewer.swift` | `withAnimation(.spring())` ×2 — double-tap zoom toggle + pinch-end recenter | `withMotion(reduceMotion, .spring())` — zoom stays functional (essential, user-initiated); only the decorative spring bounce is dropped |
| `Screens/SettingsScreen.swift` | `.transition(.move(edge:.bottom).combined(with:.opacity))` — toast | Currently inert (toast assignment has no animation transaction). Made RM-aware defensively: `reduceMotion ? .opacity : .move(...)` — forward-safe if a future `withAnimation` wrap is added |

Mechanism: new `Core/ReduceMotion.swift` `withMotion(_:_:_:)` helper —
`reduceMotion ? body() : withAnimation(animation) { body() }`. Each view
reads `@Environment(\.accessibilityReduceMotion)` (SwiftUI-native,
reactive binding to `UIAccessibility.isReduceMotionEnabled`; the
SwiftUI-only counterpart of the Compose `ReduceMotionDetector` —
no Kotlin bridge needed for pure-SwiftUI views).

Out of scope (correctly untouched, not motion violations): pure
`.transition(.opacity)`-only toasts with no animation transaction
(`DiscoveryScreen.swift` forkResultToast, `ChartViewerScreen.swift`) —
no `withAnimation` ⇒ instant ⇒ already RM-safe; a cross-dissolve is
itself Apple's recommended RM substitution. Stock SwiftUI `.sheet` /
navigation transitions auto-substitute under RM at the UIKit layer.

Remaining A25 work: none code-side. The audit row is fully closed; the
2 alpha-tester drills below (A23 + A24) are unrelated.

## A26 — Touch targets

### Code-level baseline

Material 3 IconButton default size: 48dp. SwiftUI Button + Image-based buttons: 44pt by default. Skeinly's app code does not override `Modifier.size` on IconButton anywhere outside the ChartEditor cell grid.

### Known violator (deferred)

- **ChartEditor cell touch targets**: cells render at `min(canvasW/gridW, canvasH/gridH)` px. On large grids (16×16+, polar), cells drop below 24dp. Per CLAUDE.md Tech Debt → Accessibility deferrals → "ChartEditor zoom + WCAG 2.5.8" — reopen post-beta if testers complain. Not a launch blocker because the default 8×8 grid renders ~40dp cells on a typical phone.

### Pre-alpha verdict

No other touch-target violations expected. The ChartEditor issue is documented and intentional defer.
