# ADR-003: オフラインファースト同期戦略

> Source: [English version](../../en/adr/003-offline-sync-strategy.md)

## ステータス

承認済み

## コンテキスト

編み物をする人は接続が不安定な環境（リビング、カフェ、編み物教室）で作業することが多い。アプリはコアアクティビティ（パターン閲覧、進捗追跡）においてオフラインでも機能し、接続が利用可能な場合に Supabase と同期する必要がある。

主な疑問点：
- 信頼できるソースは何か：ローカルデータベースか Supabase か？
- オフライン変更の同期時にコンフリクトをどう処理するか？
- どのデータがオフラインで利用可能である必要があるか vs. オンデマンドでフェッチするか？

## 決定

SQLDelight をオフライン永続化に使用した **Supabase ファースト + ローカルキャッシュ** 戦略を採用する。

### アーキテクチャ

```
┌─────────────────────────────────────────────┐
│                  UI レイヤー                  │
├─────────────────────────────────────────────┤
│             リポジトリレイヤー                  │
│  ┌─────────────┐     ┌──────────────────┐   │
│  │  SQLDelight  │◄───►│  Supabase Client │   │
│  │（ローカルキャッシュ）│  │（信頼できるソース）│   │
│  └─────────────┘     └──────────────────┘   │
├─────────────────────────────────────────────┤
│        Sync Manager（Coroutines）            │
│  - 接続状態の監視                              │
│  - オフラインミューテーションのキューイング         │
│  - コンフリクト解決（Last-Write-Wins）          │
└─────────────────────────────────────────────┘
```

### ルール

1. **信頼できるソース**: Supabase がすべての共有/公開データの信頼できるソース。SQLDelight は同期されるまでのオフラインドラフトの信頼できるソース。
2. **読み取りパス**: リポジトリは最初に SQLDelight から読み取り（即時 UI）、次に Supabase からフェッチして更新。Supabase Realtime サブスクリプションがローカルキャッシュに更新をプッシュ。
3. **書き込みパス**: ミューテーションは即座に SQLDelight に書き込み（楽観的 UI）、その後 Supabase への同期をキューに入れる。オンラインの場合、同期は即時。オフラインの場合、接続回復時にリトライ。
4. **コンフリクト解決**: `updated_at` タイムスタンプに基づく Last-Write-Wins。進捗追跡（行カウンター）では、より大きい段数が優先（通常の使用では編み物を解くことはない）。
5. **オフライン利用可能データ**: ユーザー自身のパターン、アクティブなプロジェクト、最近の進捗。共有パターンは初回閲覧時にキャッシュ。
6. **オンラインのみのデータ**: アクティビティフィード、コメント、検索結果、他のユーザーのプロフィール。

### SQLDelight ローカルスキーマ

Supabase スキーマのサブセットをローカルに保存：

- `Pattern`（所有パターンとキャッシュされた共有パターン）
- `Project`（ユーザーのアクティブなプロジェクト）
- `Progress`（最近の進捗エントリ）
- `PendingSync`（同期待ちのオフラインミューテーションキュー）

`PendingSync` テーブルの追跡内容：
| フィールド | 型 | 説明 |
|-----------|------|------|
| id | INTEGER (PK) | 自動インクリメント |
| entity_type | TEXT | `pattern` / `project` / `progress` |
| entity_id | TEXT | エンティティの UUID |
| operation | TEXT | `insert` / `update` / `delete` |
| payload | TEXT | JSON シリアライズされたエンティティデータ |
| created_at | INTEGER | エポックミリ秒 |
| retry_count | INTEGER | 同期試行回数 |

## 結果

### ポジティブ

- **レスポンシブ UI**: ローカルキャッシュからの読み取りにより、ネットワーク状態に関係なく即時表示を保証。
- **オフライン進捗追跡**: 接続がなくても段数を増やしたりメモを追加可能。
- **耐障害性のある同期**: キューに入れられたミューテーションはアプリ再起動後も残り、接続回復時に同期。
- **リアルタイム更新**: Supabase Realtime サブスクリプションが共有コンテンツのローカルキャッシュを最新に保つ。

### ネガティブ

- **二重ストレージの複雑さ**: SQLDelight と Supabase 間の整合性維持には、同期マネージャーの慎重な実装が必要。
- **コンフリクトのエッジケース**: 二人のユーザーが同じ共有パターンをオフラインで同時に編集した場合、Last-Write-Wins でデータが失われる可能性がある。使用パターンを考慮すると MVP では許容範囲（パターンは通常一人が編集する）。
- **ストレージオーバーヘッド**: ローカルデータベースがサーバーデータの一部を複製するため、アプリのストレージ使用量が増加。

### ニュートラル

- SQLDelight スキーマは `shared/src/commonMain/sqldelight/` 配下の `.sq` ファイルで定義。
- 同期マネージャーは shared モジュールで Kotlin Coroutines ベースのサービスとして実装。
- 初期 MVP では完全な同期キューを実装せず簡素化する可能性あり（ローカル読み取りキャッシュ付きのオンラインのみのミューテーション）。後のイテレーションでオフライン書き込みサポートを追加。

## 検討した代替案

| 代替案 | メリット | デメリット | 不採用の理由 |
|--------|---------|-----------|-------------|
| Supabase のみ（ローカルキャッシュなし） | シンプルなアーキテクチャ、単一の信頼できるソース | オフラインでアプリが使用不可、接続が悪い場合に遅い UI | 様々な接続環境で作業する編み物ユーザーにとって許容できない UX |
| ローカルファースト + 結果整合性同期（CRDT） | 強力なオフラインサポート、自動コンフリクト解決 | 実装の複雑さが大きい、確立された KMP CRDT ライブラリなし、このデータモデルには過剰 | 複雑さが不当。編み物データには Last-Write-Wins で十分 |
| SQLDelight を唯一の信頼できるソース + 定期バックアップ | シンプル、完全オフライン | リアルタイム共有なし、マルチデバイス同期なし、ソーシャル目的を損なう | アプリの共有とインタラクションというコアバリューと矛盾 |
