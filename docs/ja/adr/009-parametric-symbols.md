# ADR-009: パラメトリックシンボル契約

> Source: [English version](../../en/adr/009-parametric-symbols.md)

## ステータス

Accepted

## コンテキスト

Phase 30.x のシンボルカタログには、構造の異なる 2 系統の「数を持つ」
グリフが共存している:

1. **ID に数が埋め込まれたもの**: `jis.crochet.dc-cluster-3`,
   `jis.crochet.dc-cluster-5`, `jis.crochet.picot-3`。その個数専用に
   幾何形状が描かれており、個数が変われば ID も変わる。
2. **セルパラメータとして数を持つもの**: `jis.crochet.ch-space` は
   `parameterSlots = [ParameterSlot(key = "count", defaultValue = "n", ...)]`
   を宣言し、個数に依存しない形状の上にビューアが
   `cell.symbolParameters["count"]` をオーバーレイする。

Phase 30.4 のバックログは `picot-N` (4/5/6 目の鎖で作る長いピコット) と
`hdc-cluster-5` を要求している。明示ルールがないと新グリフ追加のたびに
「ID 埋め込み vs. パラメータスロット」の議論が蒸し返される。また
Phase 32 エディタ MVP はパラメータ入力 UI を持たないため、どのパラメトリック
セルも `symbolParameters = emptyMap()` となり、結果として `defaultValue`
のリテラル (例: `"n"`) が表示される。

Phase 32 のハンドオフはこれを実装ブロッカーとして指摘し、Phase 30.4
グリフバンドルに先行して `picot-N` を独立ミニ ADR に切り出すことを要求した。
最初のパラメトリックグリフがユーザに届く前に契約を固め、場当たり的な
ロックインを避けるためである。

本 ADR はその契約を固定する。パラメータ値入力のエディタ UI は定めない
(Phase 32.3 へ)。新規グリフの追加もしない (Phase 30.4 へ)。

## 決定

### 1. 2 軸での分類

数を持つグリフは独立な 2 軸で分類する:

- **幾何軸**: 個数が変わると描画形状も変わるか? (変わる = `geometry-varying`、
  変わらない = `geometry-invariant`)
- **ラベル軸**: 個数がセル内部に数字として描画されるか? (描画される =
  `labeled`、描画されない = `unlabeled`)

各象限の正典エントリ:

| | labeled | unlabeled |
|---|---|---|
| **geometry-varying** | `jis.crochet.picot-{N}` (N ごとに別 ID) | `jis.crochet.dc-cluster-{3,5}` |
| **geometry-invariant** | `jis.crochet.ch-space` (`count` スロット) | *パラメトリックではない — 通常グリフとして投入* |

### 2. パラメトリック = geometry-invariant かつ数字を描画

シンボルが **パラメトリック** (`parameterSlots` を使う) なのは、幾何形状が
不変で、かつ編み図の慣習として呼び出し側の値をその上に描画する場合のみ。
それ以外の「数を持つ」グリフは **離散ファミリ** (N ごとに別 ID) とする。

これで `ch-space` (パラメトリック。正解) と `dc-cluster-3` / `dc-cluster-5`
(離散ファミリ。正解 — それぞれに調整されたストローク位置を持ち、編み図は
数字をオーバーレイしない) の解釈争いが決着する。

### 3. 新規グリフのルール

N 番目のグリフを追加するときは、次の順で問う:

1. **描画は N で変わるか?** 変わる → 離散ファミリ。同じ語幹を共有する
   `{stem}-{N}` として、サポートする N ごとに 1 件ずつ `SymbolDefinition`
   を投入する。商用パターンで実際に使われる N を対象とし、あらゆる
   整数を先回りしない。
2. **セル内部に数字が描画されるか?** 描画される → パラメトリック。
   `parameterSlots` 付きの 1 件を投入する。
3. それ以外 → 通常 (非パラメトリック) グリフ。

### 4. 離散ファミリの ID 構文

- ID の形: `{namespace}.{stem}-{N}`。サフィックスは 10 進整数。
  例: `jis.crochet.dc-cluster-3`, `jis.crochet.picot-4`。
- 語幹は N に依存しない (複数形や時制を変えない)。
- 各ファミリメンバは独立したファーストクラスのカタログエントリ —
  固有のラベル、description、aliases、jisReference、pathData を持つ。
  コピー & ペーストで構わない。カタログは DRY より明示性を優先する。
- Aliases に語幹 (例: `picot`) を含めてよいのは、辞書検索で裸語が
  マッチするようにする目的で、ファミリの代表メンバ 1 件のみ
  (慣習: 最短のもの。つまり `picot-3` に `picot` を aliases として持たせる)。

### 5. パラメトリックシンボルの契約

- `ParameterSlot.key` — 安定な不透明文字列。スロット意味論をまたいで
  再利用しない。新しい意味 (例: カラー ID) は新しいキーにする。
- `ParameterSlot.defaultValue` — 任意。非 null でセルがそのキーを
  持たない場合、ビューアはリテラルとして default を描画する。
  空の `ch-sp` が "n" を表示したい作者はこれに依存する。
- **値の型は `String`。** `ChartCell.symbolParameters` はワイヤ上
  `Map<String, String>`。数値バリデーション (もしあれば) は Phase 32.3
  エディタ UX の範疇で、ドメインモデル側では強制しない。
  根拠: 編み図は最終的にグラフィックであり、`"n"`, `"x"`, `"?"` は
  編み手が実パターン中に使う正当なプレースホルダ文字である。
- パラメトリックシンボルの幾何形状は **パラメータ値に依存してはならない**。
  レンダラはスロットアンカの上に stringify 済みの値を重ねるだけで、
  pathData を再解析しない。幾何が変わる必要があるなら離散ファミリに
  分割する (§4)。
- `ChartCell.symbolParameters` に `SymbolDefinition` が宣言していない
  未知キーがあっても、ラウンドトリップ時に保持され、レンダラは無視する。
  前方互換: 将来カタログ版がスロットを追加しても旧セルはデータ欠損なく
  復号できる。

### 6. コンテンツハッシュ不変条件

- `computeContentHash` は既に各 `ChartCell` の完全 JSON シリアライゼーション
  経由で `symbolParameters` を含む。同じ描画でパラメータ値だけ異なる 2 件は
  別ハッシュとなる。これは正しい: 描画結果が違うので、同期は別物として
  扱うべきである。
- ADR-008 §Phase 32.1 は `craftType` / `readingConvention` を
  ハッシュから外した。あれらは描画の *外* のメタデータだからである。
  パラメータ値は描画の *内* なのでハッシュに含む。

### 7. エディタ UX デルタ (Phase 32.3 の範疇)

Phase 32 MVP エディタはすべてのセルで `symbolParameters = emptyMap()` を
書き込む。そのためパラメトリックセルではビューアが `defaultValue` に
フォールバックし、プレースホルダがそのまま表示される。現状、ユーザは
ドキュメントを手動編集する以外に回避手段を持たない。

Phase 32.3 で追加する:

- **配置フロー**: パレット選択の `parameterSlots` が非空なら、
  タップ配置イベントはセルを即確定せず、`ParameterSlot.enLabel` /
  `ParameterSlot.jaLabel` をキーに小さなインライン入力 (Compose
  `BasicTextField` / SwiftUI `TextField`) を開く。
- **確定 / キャンセル**: 確定で `symbolParameters = { slot.key -> 入力値 }`
  を書き込み、キャンセルで破棄。
- **再編集**: 既存のパラメトリックセルをタップすると、現在値を初期値に
  した同じインライン入力が開く。
- **離散ファミリは影響なし** — `dc-cluster-5` のパレットエントリを
  タップしたらその ID でセルを置くだけ。プロンプトなし。
- **スキーマ変更なし** — `ChartCell.symbolParameters` は既に存在し、
  今日時点でラウンドトリップする。

Phase 32.3 自体は本 ADR の範疇外。本 ADR が固定するのはエディタが従う
べき形だけであり、picot-N バンドルと今後のパラメトリックグリフが
エディタの想定外で投入されないようにする。

### 8. `picot-N` の結論 (本 ADR のトリガ)

§3 に照らせば、`picot-N` は明らかに N で幾何形状が変わる — 3 目鎖ピコットは
小さいループ、5 目鎖ピコットは高いループまたは弧、6 目鎖ピコットは
2 セルにまたがり得る。したがって離散ファミリとする:

- Phase 30.4 で `jis.crochet.picot-{4,5,6}` を既存 `picot-3` に並べて
  追加する。
- それぞれにループサイズに合わせた `pathData` を持たせる。既存
  `picot-3` の ID は安定。
- `picot-3` に alias `picot` を付与し、裸語の辞書検索を維持する。
- `picot-5` / `picot-6` はループがセルを自然にはみ出すなら
  `widthUnits = 2` にしてよい (グリフごとに knitter アドバイザリで決定)。
- どのメンバも `parameterSlots` を持たない。

`picot-N` を **パラメトリックにしない** のは、編み図慣習が数字を描画
しないためである — ループサイズそのものが信号である。

### 9. 反対ケース: `turning-ch`

既存の `jis.crochet.turning-ch` は鎖オーバルを縦に 3 つ積んだグリフで、
立ち上がりの鎖目数 (典型 1..6) を問わず使われる。Phase 30.2 knitter
アドバイザリ §Q8 は `turning-ch-2 .. turning-ch-6` を Phase 30.3 に
先送りしたが、未実装のままである。

本 ADR の下では `turning-ch` は **パラメトリックグリフ候補** となる
(鎖目数は一般的な JA 編み図慣習ではセル内に描画される数字である)。
Phase 30.4 は次のいずれかを選べる:

- 既存の三重オーバルを保持 (「立ち上がりの鎖」を示す幾何不変の視覚信号)
  し、`count` スロットを追加して作者が数字をオーバーレイできるようにする。
  **推奨。**
- `turning-ch-{2..6}` の離散ファミリに分割。オーバルの積み段数が N で
  変わる形。一部出版社にはより忠実だがカタログエントリが 5 倍になり、
  既存の描画と噛み合わない。

Phase 30.4 がこのグリフを扱うときの判断は `knitter` エージェントに任せる。
どちらの経路も本 ADR は許容する。

## 帰結

### ポジティブ

- パラメトリック vs. 離散ファミリの判断が機械化される (§3)。
  グリフごとに蒸し返されない。
- `picot-N` は離散ファミリとして投入でき、既存 `picot-3` の ID は安定。
  2 軸ルールがその選択を正当化する。
- `ch-space` のパラメトリック形は正しいままと確認された。
- `ChartCell.symbolParameters` がパラメータ値の正式なオーサリング面として
  確定し、Phase 32.3 エディタ UX の実装を解禁する。
- `turning-ch` が本 ADR 内で意思決定を強制されずに解決経路を得た。

### ネガティブ

- 離散ファミリはカタログエントリが重複する (実質的に似たグリフを N 個コピー)。
  予想される N の小ささ (picot で 3..6、dc-cluster で 3/5) と、N 個別に
  幾何を調整できる価値を踏まえ許容する。
- パラメトリックシンボルは Phase 32 MVP のうちは `defaultValue` を
  そのまま描画する。`ch-space` を作成するユーザには Phase 32.3 が出るまで
  プレースホルダ "n" が表示される。既知事項であり修正対象ではない。

### 中立

- `widthUnits` / `heightUnits` は引き続きエントリ単位の値なので、
  複数セルにまたがる離散ファミリメンバ (例: 仮想的な `picot-6`) は
  N とは独立に span を宣言する。

## 検討した代替案

| 代替案 | 不採用の理由 |
|---|---|
| `count` スロット付きの単一パラメトリック `picot` | N で幾何が変わる必要がある — 6 目鎖ループは 3 目鎖ループの拡大ではなく、編み図慣習はサイズで読むもので数字を描画しない。§2 違反。 |
| 数を持つグリフを全てパラメトリックに強制 | `dc-cluster-3/5` の N 個別幾何調整が失われる。これらファミリの編み図慣習は数字をオーバーレイしない。 |
| 数を持つグリフを全て離散ファミリに強制 | `ch-space` は想定スパンごとに `ch-space-2`, `ch-space-3`, ... を要し無限列挙となる。編み図慣習が *数字を描画する* のにその利点が死ぬ。 |
| 整数型 `symbolParameters: Map<String, Int>` | 実パターンは `n`, `x`, `?` のような非数値プレースホルダを使う。文字列型の方が描画成果物に近い。 |
| パラメトリック ID 正規表現をゲート (例: 末尾 `-N` 必須) | 直交する 2 軸 (幾何とラベル) を混同する。§1 の 2 軸テーブルが正しい因数分解。 |
