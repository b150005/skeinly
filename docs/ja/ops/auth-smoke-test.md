# 認証スモークテストランブック

> Phase 26.8 (ADR-022 §6.7) — Phase 26 で構築した認証面（OAuth、MFA、
> 生体認証、linkIdentity マージ）をクローズドベータのテスター招待を
> 送る前に実機で end-to-end 手動検証するためのランブック。
>
> Phase 26.x のスライスをシップするたび、およびクローズドベータの
> 最初のテスター招待を送る前に実行する。リリース候補ビルドが上がる
> たびに再実行する。
>
> **対象**: 運用担当者（プロジェクトオーナー）。**テスター向けではない。**
> テスターには別途ベータ評価表（[phase-39-beta-rubric.md](../phase/phase-39-beta-rubric.md)）
> を渡す。
>
> 詳細はそれぞれの英語版 [docs/en/ops/auth-smoke-test.md](../../en/ops/auth-smoke-test.md) を
> 参照。本ドキュメントはチェックリスト形式の翻訳。

## 前提条件

- iOS 17 以上の実機（TestFlight ビルド OR Xcode によるローカル署名）。
- API 26 以上の Android 実機（Play 内部テストビルド OR ローカル署名
  デバッグ APK）。
- Apple Sandbox テスターアカウント（App Store Connect → ユーザーと
  アクセス → Sandbox テスター から作成）。
- Google テストアカウント（一般的な gmail.com アカウントで可。
  OAuth Sandbox は自動的に有効）。
- 認証アプリ（Google Authenticator / 1Password / Authy 等、任意）。
- Apple + Google OAuth プロバイダーを設定済みの Supabase プロジェクト
  （Dashboard → Authentication → Providers）。

## 1. メール / パスワードのベースライン

iOS + Android 両方で実行。

1. クリーンインストール。
2. サインアップ画面で `smoke-test-<時刻>@example.com` + 16 文字パスワード。
3. アカウント作成成功 → ProjectList 到着を確認。
4. 設定 → アカウント → サインアウト。
5. 同じ資格情報でサインイン。設定 → アカウント にメール表示確認。

**期待結果**: ✅ 両プラットフォーム。

## 2. Apple サインイン（iOS のみ）

1. クリーンインストール OR §1 からサインアウト。
2. Login 画面で「Apple でサインイン」をタップ。
3. SignInWithAppleButton 表示。Sandbox テスター Apple ID を選択。
   初回サインイン時に Apple がメール共有方法を尋ねるので、
   **プライベートリレーを選択**（リレー開示コピーをテストするため）。
4. **OAuth プロフィール設定ゲート** が表示されることを確認：
    - タイトル「あなたの名前を教えてください」。
    - 名前フィールドが Apple 由来の名前で pre-fill されている
      （初回のみ。以降は Apple が `full_name` クレームを再送
      しないためゲートはスキップされる）。
    - アバタータイルなし（Apple は picture URL を公開しない）。
    - 「保存」タップ → ProjectList へ。
5. 設定 → アカウント行を確認：
    - メールが `<ランダム>@privaterelay.appleid.com` 形式。
    - 「サインイン方法: Apple (プライベートリレーメールを使用)」コピー。
6. サインアウト → 同じ Apple ID で再サインイン → プロフィール設定
   ゲートが表示されない（preference flag が再表示を防止）→
   ProjectList へ直接到着。

**期待結果**: ✅ iOS のみ。

## 3. Android での Apple サインイン（web-OAuth + Custom Tabs）

1. Android クリーンインストール OR サインアウト。
2. Login 画面で「Apple でサインイン」をタップ。
3. Custom Tab が Supabase ホスト OAuth フロー → Apple サインイン Web
   ページ → `skeinly://auth-callback` へ戻る。
4. §2.4 と同じプロフィール設定ゲートを確認（Android 変種）。
5. 設定 → アカウント行が §2.5 と一致することを確認。

**期待結果**: ✅ Android のみ。

## 4. Google サインイン

iOS + Android 両方で実行。

1. クリーンインストール OR サインアウト。
2. 「Google でサインイン」タップ。
3. Google テストアカウントを選択。
4. **OAuth プロフィール設定ゲート** を確認：
    - 名前フィールドが `user_metadata.name` から pre-fill。
    - **アバタータイル表示**。「この画像をアバターとして使いますか?」
      ラベル。「この画像を使う」タップ → スピナー → 「画像を取り込み
      ました」コピー + 「別の画像を選ぶ」アフォーダンス。
5. 「別の画像を選ぶ」をタップ。ヒントコピーが表示されることを確認：
   「画像はあとで 設定 → プロフィール からいつでも変更できます。」
   （Phase 26.7 Tech Debt 解消）。avatar-imported 状態がリセットされ、
   再度「この画像を使う」をタップすれば復元できることを確認。
6. 「保存」タップ → ProjectList へ。
7. 設定 → アカウント行を確認：
    - メールが Google アドレス。
    - 「サインイン方法: Google」コピー。
8. 設定 → プロフィール を確認：取り込んだアバターが表示されている。

**期待結果**: ✅ 両プラットフォーム。

## 5. linkIdentity マージフロー（同じメールでメール / パスワード + OAuth）

1. クリーンインストール。`linktest@example.com` + パスワードで
   サインアップ。
2. サインアウト。
3. `linktest@example.com` と同じメールの Google アカウントで
   「Google でサインイン」をタップ。
4. LinkIdentityRequired ダイアログ表示：「このメールアドレスはすでに
   使用されています — Google を既存アカウントにリンクするには
   パスワードを入力してください」を確認。
5. 手順 1 のパスワードを入力 → 「アカウントをリンク」。
6. ProjectList 到着確認。設定 → アカウント に「サインイン方法:
   メールアドレスとパスワード」が表示され、「リンク済みアカウント」
   セクションに「Google」が含まれることを確認。

**期待結果**: ✅ 両プラットフォーム。

## 6. TOTP MFA 登録 + チャレンジ

片方のプラットフォームで実行（Phase 26.5 のロジックは shared モジュー
ル、UI 面は両方とも対称）。

1. サインイン → 設定 → セキュリティ → 「2要素認証を有効にする」。
2. QR コード（または base32 文字列）を認証アプリでスキャン。画面に
   表示される 16 文字の復旧コードを保存（手順 8 で使用）。
3. 認証アプリの現在の 6 桁コードを入力 → 「検証」。
4. 設定 → セキュリティ行に「2要素認証: オン」表示確認。
5. サインアウト → メール / パスワードでサインイン。
6. **TOTP チャレンジ画面** が表示されることを確認。6 桁コード入力 →
   ProjectList 到着。
7. サインアウト → 再サインイン → 「復旧コードを使う」タップ → 手順 2
   の復旧コードを入力。
8. ProjectList 到着確認。設定 → セキュリティ行が「2要素認証: オフ」
   に変わっている（復旧コード消費 + factor 解除）。
9. 手順 1 から TOTP を再登録。

**期待結果**: ✅ 最低 1 プラットフォーム、もう片方はスポット確認。

## 7. 生体認証による再認証

iOS + Android 両方で実行。OS 設定で Face ID / Touch ID（iOS）または
指紋 / 顔（Android）を登録済みの実機が必要。

1. サインイン → 設定 → セキュリティ → 「生体認証」。
2. トグル ON → OS 生体プロンプトが即座に発火。成功すると ON のまま
   閾値ピッカー（1 分 / 5 分 / 15 分 / 1 時間）が表示。
3. 閾値を 1 分に設定。
4. アプリをバックグラウンド → 70 秒以上待機 → フォアグラウンド。
5. 「Skeinly に戻るには本人確認が必要です」コピーで生体プロンプト
   発火確認。プロンプトをキャンセル。
6. アプリがサインアウトされ Login 画面に戻ることを確認（ADR-022 §3.8
   — resume-cancel → サインアウト degradation）。
7. 再サインイン → 設定 → セキュリティ → 「2要素認証をオフにする」。
8. 「2要素認証をオフにするため本人確認が必要です」コピーで生体プロンプ
   ト発火確認（重要操作ゲート、フォアグラウンド再認証ゲートとは別）。
9. 設定 → アカウント → 「アカウントを削除」。
10. 「アカウントを削除するため本人確認が必要です」コピー確認。
    キャンセル → アカウント削除されないことを確認。再試行 + 承認 →
    削除完了 → Login へ。

**期待結果**: ✅ 両プラットフォーム。

## 8. OS 設定往復後の生体認証可用性リフレッシュ

Phase 26.7 Tech Debt 検証。

1. OS 設定で生体 / PIN を**未登録**にした端末でサインイン
   （iOS 設定 → Face ID とパスコード → パスコードオフ、または
   Android 設定 → セキュリティ → 画面ロック → なし）。
2. 設定 → セキュリティ → 生体認証 を開く。
3. ステータス行「OS設定で生体認証または PIN を登録してください」
   表示・トグル無効 を確認。
4. OS 設定に切り替え、生体 / PIN を登録、Skeinly に戻る。
5. 画面が `onAppear`（iOS）/ `LaunchedEffect`（Android）で自動的に
   可用性を再クエリし、トグルが有効化されることを確認。ステータス
   行が「オフ」（= トグル準備完了）に更新される。

**期待結果**: ✅ 両プラットフォーム。

## 9. アカウント削除カスケード

1. §1 のメール / パスワードアカウントでサインイン。§6 で TOTP 登録。
   §7 で生体認証有効化。
2. 設定 → アカウント → 「アカウントを削除」 → 生体プロンプト承認 →
   削除確定。
3. Supabase Dashboard で次を確認：
    - `auth.users` 行が消えている。
    - `profiles` 行が消えている（ON DELETE CASCADE）。
    - `device_tokens` 行が消えている（ON DELETE CASCADE）。
    - `mfa_recovery_codes` 行が消えている（ON DELETE CASCADE）。
    - `subscriptions` 行が消えている（ON DELETE CASCADE）。
4. 同じメールでサインインを試みる → "User not found" でエラー。

**期待結果**: ✅ Supabase Dashboard で確認済み。

## A0d-6 Data Safety 再申告レシピ

§1 〜 §9 すべて PASS 後、Play Console Data Safety を更新：

1. Play Console → すべてのアプリ → Skeinly → アプリのコンテンツ →
   データセーフティ → 管理。
2. ページ 3「収集されるユーザーデータの種類」：
    - **写真と動画** → 「写真」にチェック（以前は未チェック。OAuth
      アバター取り込みでユーザー写真が Storage に保存されるためオン
      に切り替え）。「動画」はまだ未チェック（Phase 28 未着手）。
    - **個人情報** → 「名前」にチェック（OAuth + プロフィールで
      常に収集）。
    - **個人情報** → 「メールアドレス」にチェック（既にチェック済み
      のはず）。
3. ページ 4「アカウントの作成方法」：
    - 「OAuth」にチェック（Phase 26.1 + 26.2 で追加）。
    - 「ユーザー名、パスワード、その他の認証」にチェック（Phase 26.5
      で TOTP がシップされたため適用可能になった）。
4. 保存 → 申告 → 次の Play リリースに反映。レビュー状態が「アクティブ」
   から「審査中」（1〜3 日）→「アクティブ」に戻る。

Apple App Store Connect プライバシー栄養成分表は並行サーフェス —
「App プライバシー」 → 編集 → 同じ更新を反映。Apple の審査は通常
早く（多くの場合当日中）。
