# ECC による TDD ワークフロー

## TDD とは？

テスト駆動開発（TDD）は、実装コードを書く前にテストを書くソフトウェア開発方法論です。これにより、すべてのコードに対応するテストが存在し、実装が実際の要件に基づいて駆動されることが保証されます。

## RED-GREEN-IMPROVE サイクル

### 1. RED — 失敗するテストを書く

期待される動作を記述するテストを書きます。実行します。実装がまだ存在しないため、失敗するはずです。

```
# 疑似コード
test "calculate_total はアイテム価格の合計を返す" {
  items = [{ price: 100 }, { price: 200 }]
  result = calculate_total(items)
  assert result == 300
}
```

**なぜ RED から始めるのか？** 最初から成功するテストは、意図した内容をテストしていない可能性があります。まず失敗を確認することで、テストが意味のあるものであることを確認できます。

### 2. GREEN — 最小限の実装を書く

テストを通過させる最もシンプルなコードを書きます。追加機能、最適化、エッジケースの処理はまだ加えません。

```
# 疑似コード
function calculate_total(items) {
  sum = 0
  for item in items {
    sum = sum + item.price
  }
  return sum
}
```

### 3. IMPROVE — リファクタリング

テストが通過したので、動作を変えずにコード構造を改善します。テストがセーフティネットとして機能します — リファクタリング後もテストが通過すれば、何も壊れていません。

```
# 疑似コード（関数型スタイルにリファクタリング）
function calculate_total(items) {
  return items.reduce((sum, item) => sum + item.price, 0)
}
```

## ECC エージェント: tdd-guide

`tdd-guide` エージェントは TDD ワークフローを自動的に強制します：

1. 新機能をリクエストすると、まずテストの雛形を書きます
2. テストを実行して失敗を確認します（RED）
3. テストを通過させる最小限のコードを実装します（GREEN）
4. リファクタリングの機会を提案します（IMPROVE）
5. カバレッジが 80% の閾値を満たしているか検証します

このエージェントは、Claude Code が機能リクエストやバグ修正を検出した場合に自動的に呼び出されるか、`/tdd` で手動で呼び出されます。

## テストの種類

### 単体テスト

個々の関数やモジュールを分離してテストします。

- 外部依存関係（データベース、API、ファイルシステム）をモック化します
- 高速実行（テストごとにミリ秒単位）
- 高いカバレッジ目標：行カバレッジ 80% 以上を目指します

### 結合テスト

コンポーネントが連携して動作するかテストします。

- 可能な限り実際のデータベース（モックではなくテストインスタンス）を使用します
- API エンドポイントのリクエスト/レスポンスサイクルをテストします
- データベースクエリが期待される結果を返すか検証します

### E2E テスト

完全なユーザーフローを最初から最後までテストします。

- ブラウザ自動化を使用します（Playwright 推奨）
- 重要なパスをカバーします：ログイン、コア機能、チェックアウトなど
- 低速ですが、単体/結合テストが見逃す問題を検出します

## カバレッジ要件

このテンプレートは最低 **80% のテストカバレッジ** を強制します。これは CI/CD パイプラインで設定され、すべてのプルリクエストでチェックされます。

カバレッジは以下で測定されます：
- **行カバレッジ**: テストによって実行されたコード行の割合
- **ブランチカバレッジ**: テストされた条件分岐の割合

## ベストプラクティス

1. **テストごとに1つのアサーション** — 各テストは1つの特定の動作を検証すべきです
2. **説明的なテスト名** — テスト名は実装ではなく、期待される動作を記述すべきです
3. **テストの分離** — テストは互いに依存したり、変更可能な状態を共有したりすべきではありません
4. **Arrange-Act-Assert** — 各テストをセットアップ、実行、検証のフェーズで構造化します
5. **実装を修正し、テストは修正しない** — テストが失敗した場合、通常は実装が間違っています（テスト自体にバグがない限り）

## トラブルシューティング

テストが予期せず失敗した場合：

1. テストの分離を確認 — テスト間で共有状態が漏れていないか？
2. モックを検証 — 期待される値を返しているか？
3. テスト環境を確認 — テストデータベースは正しくシードされているか？
4. 構造化されたデバッグ支援のために `tdd-guide` エージェントを使用してください
