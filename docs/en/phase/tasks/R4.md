# R4 — ProjectDetail Dynamic Type

## Status

`READY_FOR_CONSOLIDATION`

## Scope

Replace ProjectDetail counter + +/- buttons' fixed sizes with scale-respecting Material 3 typography (Compose) and `@ScaledMetric` (SwiftUI). Concretely: rowCounter `fontSize=96.sp` (KT) / `DesignTokens.counterFontSize` (SW), decrement/increment buttons `Modifier.size(72.dp)`/`size(96.dp)` containers + inner `fontSize=28.sp`/`36.sp` (KT) and `DesignTokens.decrementButtonSize`/`incrementButtonSize` (SW). Counter container is allowed to grow at any Dynamic Type level (audit "let containers grow" — no clamp). `DesignTokens.swift` values stay as the base; `@ScaledMetric` scales them through `UIFontMetrics` without editing the tokens file. Out of scope: `DesignTokens` value re-definition, pattern/discovery list `maxLines=1`/`.lineLimit(1)` relaxation (no such clamp exists on ProjectDetail itself per exploration), 18.dp status-badge icons, Dynamic Type cap policy (escalated to orchestrator via Follow-ups).

**What shipped (R4 — ProjectDetail Dynamic Type)**:

- Compose `ProjectDetailScreen.kt` `CounterSection`: rowCounter `fontSize=96.sp` → `MaterialTheme.typography.displayLarge`; decrement / increment `Modifier.size(72.dp/96.dp)` → `Modifier.defaultMinSize(...)`; inner `-`/`+` `fontSize=28.sp/36.sp` → `headlineMedium`/`displaySmall`. `defaultMinSize` import added; unused `…unit.sp` import removed. `testTag`s + i18n / Maestro touch-paths unchanged.
- SwiftUI `ProjectDetailScreen.swift`: 3 `@ScaledMetric` properties added on the `ProjectDetailScreen` struct (counter relativeTo `.largeTitle`, decrement/increment relativeTo `.title`) reading `DesignTokens.{counterFontSize, decrementButtonSize, incrementButtonSize}` as base values; the 3 inline `.font(.system(size: DesignTokens.…))` call sites now read the scaled variants. `DesignTokens.swift` untouched (write-set discipline).

**Key design decisions**:

- **No Dynamic Type clamp applied.** Audit directs "let the counter/+− containers grow"; app-wide cap policy escalated to Follow-ups for orchestrator review (recommend dedicated ADR slot — see Follow-ups).

- **`Modifier.defaultMinSize` over inner-content padding.** Keeps WCAG 2.5.5 touch-target minimums (72×72 / 96×96 dp at fontScale 1.0) while allowing growth at fontScale ≥ 1.0.

## Follow-ups

- **Dynamic Type cap policy**: audit and R4 row do not prescribe a clamp (`.dynamicTypeSize(...DynamicTypeSize.accessibility3)`). This task lets containers grow to full AX5 per audit's "let containers grow" line. Recommend orchestrator allocates an ADR slot to decide app-wide clamp range (vs. unbounded growth) before Phase-40 GA, after UI regression at AX4/AX5 is observed in pre-alpha drill (audit §A24 step).
- **Audit R4 remainder — `DesignTokens.swift` re-definition**: SW 72/48/64 pt absolute constants remain as bases. A follow-up slice should consider migrating these to relative `Font.TextStyle` references or `@ScaledMetric`-bearing computed properties at the tokens layer (rather than inline at every consumer).
- **Audit R4 remainder — pattern/discovery list description clamps**: PatternListScreen / DiscoveryListScreen `maxLines=1` (KT) / `.lineLimit(1)` (SW). Not present on ProjectDetail itself per exploration; separate slice.
- **R5 polish candidate**: ProjectDetailScreen.kt L878/L888/L914 `Modifier.size(18.dp)` Icon (status badge / reset / overflow). Outside §4 R4 "counter/+− containers" wording, route into R5 state-not-color polish or a dedicated icon-scale slice.
