## CI Known Limitations

> **Format discipline (per [docs/process/upstream-issue-triage.md](docs/process/upstream-issue-triage.md))**: every new entry below MUST link to its upstream tracking issue, carry a `last-checked` date, name the applied workaround if any, and set a `Re-check by` date (default monthly). Workarounds are explicitly temporary — when upstream closes the issue, the workaround commit removes itself. Entries that predate this discipline (CodeQL java-kotlin, CMP resources, iOS NavigationStack edge-swipe, XCUITest flake) are kept as-is and would be re-formatted only if their underlying issue resurfaces. New entries: use the [template in upstream-issue-triage.md §3](docs/process/upstream-issue-triage.md#3-track--record-the-entry--workaround-discipline).

- **CodeQL java-kotlin**: ✅ **Resolved twice** — first on 2026-05-02 (`1298b4b`) by switching to `:shared:assembleAndroidMain :androidApp:assembleDebug` over `:shared:compileAndroidMain` (which is a lifecycle task with no kotlinc invocation under `com.android.kotlin.multiplatform.library`). Then re-failed once Phase 39.3 had warmed the gradle-actions cache, because `:shared:compileAndroidMain` and `:androidApp:compileDebugKotlin` started returning FROM-CACHE — kotlinc never executed and the CodeQL extractor saw an empty database ("CodeQL detected code written in Java/Kotlin but this run didn't build any of it"). Second resolution adds `--no-build-cache` to the Build (java-kotlin) gradle invocation in `security.yml` so kotlinc is forced to run on every CodeQL workflow run regardless of cache warmth. `CodeQL Analysis (java-kotlin, ubuntu-latest, manual)` is in `required_status_checks` on the `main-strict` ruleset; `continue-on-error: true` removed. Both load-bearing pieces (assemble pipeline + `--no-build-cache`) documented inline in `security.yml`. The `github/codeql-action` floating `@v4` tag preserved so dependabot tracks CodeQL CLI patches.
- **Maestro 2.5.x iOS driver setup broken on Xcode 26.x**: host `make e2e-ios` fails with `Failed to connect to /127.0.0.1:7001` immediately after Skeinly app reinstall. Root cause: Maestro 2.5.x's CLI JAR omits `MaestroDriverLib/**` from its packaging include list, so the iOS driver framework's `Info.plist` is missing at extract time and `xcodebuild` aborts. The driver app never lands in `xcrun simctl listapps` and there is no listener on port 7001.
  - **Upstream**: [mobile-dev-inc/Maestro#3218](https://github.com/mobile-dev-inc/Maestro/issues/3218) (OPEN, last-checked 2026-05-06) — packaging bug with a community-supplied one-line fix in `maestro-cli/build.gradle.kts` (no PR yet). Related: [#3248](https://github.com/mobile-dev-inc/Maestro/issues/3248) (OPEN — Xcode 26.4 simulator path broken in Maestro 2.5.1, `list-devices` empty and `test --udid` hangs), [#3254](https://github.com/mobile-dev-inc/Maestro/issues/3254) (OPEN — driver becomes unresponsive after ~3 tests on iOS 26.4), [#3137](https://github.com/mobile-dev-inc/Maestro/issues/3137) (CLOSED 2026-04-07 — reporter confirmed Maestro 2.0.10 is the only version that runs on iOS 26 + Apple Silicon + Xcode 26.x; 2.3.0+ regressions).
  - **Affected scope**: host `make e2e-ios` on Xcode 26.x + Maestro 2.5.x + iPhone 17/iOS 26.4 simulator. CI runner (`macos-latest` GitHub Actions = Xcode 16-series + iOS 18 sim + iPhone 16) is unaffected — the same flow batch passes there, and `make e2e-ios` is gated server-side via the `main-strict` ruleset.
  - **Workaround**: none applied. Host iOS E2E is grey while CI gates the same suite. Maestro 2.0.10 downgrade (per #3137) reportedly works but loses 2.x features and reverses our position as an active upstream consumer; deferred per user direction 2026-05-06 ("active library, wait for the fix").
  - **Expected fix**: no upstream eta. The #3218 packaging fix is one-line and could land in any 2.5.x patch; #3248 / #3254 are deeper. Watch for a 2.5.2+ release.
  - **Re-check by**: 2026-06-06 (monthly). Re-run `gh issue view 3218 --repo mobile-dev-inc/Maestro` + try `make e2e-ios` against current host environment, update this entry's dates accordingly.

Closely-related anchors that are runnable (`gh issue view <num> --repo mobile-dev-inc/Maestro`) at re-check time: [#3148](https://github.com/mobile-dev-inc/Maestro/issues/3148) (CLOSED 2026-04-28, "tapOn ... reports COMPLETED but doesn't navigate (iOS 26.2)" — same shape: tap registers but UI does not respond), [#3090](https://github.com/mobile-dev-inc/Maestro/issues/3090) (CLOSED 2026-04-04, "tapOn fails to open SwiftUI Menu when placed in a ZStack overlay" — tangential SwiftUI tap surface bug). At re-check time, also re-search with keywords `"List section"` / `searchable` / `tapOn SwiftUI iOS 26` against the Maestro repo to catch any new dedicated issue that surfaced; this entry's anchors will be promoted then.
  - **Affected scope**: flows tagged `skip-ios26` (search + filter chip variants) — already excluded from `run-ios.sh` via `--exclude-tags skip-ios26`. CI XCUITest covers the same surface natively for the same screens.
  - **Workaround**: `--exclude-tags skip-ios26` already wired in `e2e/run-ios.sh`. Per-flow tag annotations remain on the affected YAMLs.
  - **Expected fix**: rolls in with broader Maestro upstream iOS 26 support — same fix wave as #3218 / #3248 cluster.

- **Maestro + SwiftUI Button on headless CI**: Maestro `tapOn` (both text and accessibility ID) does not reliably register SwiftUI `Button` taps on headless CI simulators (macOS GitHub Actions runner). The iOS onboarding helper bypasses this by relaunching the app with `-has_seen_onboarding true` launch argument, which NSUserDefaults picks up from the argument domain.
- **E2E local-only mode**: CI builds use empty Supabase credentials (local-only mode). Flows that require a live Supabase backend (e.g., Discovery browse) are tagged `requires-supabase` and excluded from CI. Run these flows locally against a Supabase-connected build.
- **iOS NavigationStack edge-swipe**: iOS 16+ `NavigationStack` does not expose a public API to disable the left-edge swipe-to-pop gesture while keeping the rest of the bar interactive. `.interactiveDismissDisabled` applies only to modally presented sheets, not pushed stack entries. Screens with discard-guard semantics (e.g. the structured chart editor) hide and replace the default back button so tapping it routes through a confirmation dialog, but the edge-swipe still pops silently. Disabling the gesture requires dropping to UIKit (`navigationController.interactivePopGestureRecognizer.delegate = self`), which is out of scope for the shared-module architecture. Users on iOS who care about in-flight edits should use the top-bar back button.
- **CMP resources on AGP 9.x KMP library plugin**: The `com.android.kotlin.multiplatform.library` plugin used by `:shared` (Phase 15.5) does not wire the Compose Multiplatform resource generator into the Android assets pipeline. The `preparedResources/commonMain/composeResources/` CVR files never get copied into `assembledResources/androidMain/`, so the final APK contains zero `composeResources/` entries and any `stringResource(Res.string.*)` throws `MissingResourceException` at runtime. Worked around in `androidApp/build.gradle.kts` via a `CopyComposeResourcesForAndroid` task wired through `androidComponents.onVariants { variant.sources.assets?.addGeneratedSourceDirectory(...) }` — resolves to `assets/composeResources/<package>/values*/*.cvr` in the APK. Remove the workaround when upstream JetBrains integration lands for this plugin. Note: Phase 33.1 shipped with this crash latent (ChartEditorScreen is never opened in E2E); Phase 33.1.1 exposed it via the SymbolGalleryScreen migration, which is exercised by `e2e/flows/android/P1_symbol_gallery.yaml`.
- **iOS XCUITest `testDeepLink_invalidURL_doesNotNavigate` simulator flake**: `DeepLinkUITests.testDeepLink_invalidURL_doesNotNavigate` has flaked once on the `macos-15-arm64` GHA runner (CI run 24885305441, commit 58395e4) with `Failed to get background assertion for target app with pid NNN: Timed out while acquiring background assertion.` This is the XCTest ⇄ SpringBoard assertion-acquisition subsystem timing out — unrelated to the app code, and reproduces as transient on cold simulators under runner contention. Does not indicate a Phase 34 regression (shared + Android jobs green on the same commit). Re-run of the same workflow would almost certainly pass. If the flake becomes chronic, the two interventions are: (1) add a retry wrapper around the XCUITest invocation at the workflow level, or (2) add `CocoaLumberjack`-style debug logging to `AppRootView` to observe the foreground-activation lifecycle the assertion acquisition is gating on. Until either trigger fires, we document and tolerate.
- **iOS Maestro `P0_create_project` simulator-restart flake**: `P0_create_project` failed once on the `macos-15-arm64` GHA runner (CI run 25600433802 attempt 1, commit 48ab4fd) with `App crashed or stopped while executing flow` — the app launch reported `COMPLETED` then the XCTestDriver `viewHierarchy` probe 583 ms later returned 500 `Application io.github.b150005.skeinly is not running`. Re-run of the same workflow on the same commit (same CI run, attempt 2) passed all 5/5 iOS flows including `P0_create_project`. Pattern: only the third iOS flow in the alphabetical run order (Maestro 2.4.0 sequence: `P0_row_counter` → `P2_navigation` → `P0_create_project`) was affected; the prior two and the subsequent two flows passed end-to-end. Each Maestro flow on iOS triggers a `clearState` cycle that uninstalls + reinstalls the app + cycles `applesimutils --setPermissions all=unset` then `all=allow` (calendar / camera / contacts / faceid / homekit / medialibrary / microphone / motion / photos / reminders / siri / speech / userTracking — notifications are NOT in the set, so `UNUserNotificationCenter` stays `notDetermined`); after ~3 successive cycles the simulator's launch services daemon (`lsd`) appears to lag, allowing `simctl launch` to return `COMPLETED` while the launched process is in fact terminating. The crash window (~600 ms) is too short for any user-visible app-lifecycle code to surface; SwiftUI scene-graph init never paints (the artifact screenshot shows a blank white screen with only the status bar). Not a Phase 24.2e regression — `@UIApplicationDelegateAdaptor(AppDelegate.self)` + `aps-environment = production` entitlement were inspected and ruled out (the same code path runs in all 5 flows; reproducibility is third-flow-specific, not code-specific). Does not indicate a Phase 24.3 regression either (the failed flow doesn't touch any push code path). If the flake becomes chronic, two interventions: (1) add a retry wrapper around the iOS Maestro step at the workflow level (mirrors the proposed XCUITest retry above), or (2) explicitly `xcrun simctl shutdown <UDID>` + `boot` between iOS Maestro flows so each flow gets a freshly-booted simulator (5–10 s overhead per flow × 5 flows = 25–50 s total cost on the iOS E2E job; acceptable if needed). Until either trigger fires, we document and tolerate.
- **Security / CodeQL `swift, macos-26` `actool` 6h hang (first occurrence, transient)**: the `CodeQL Analysis (swift, macos-26, manual)` job in Security workflow run [25979870162](https://github.com/b150005/skeinly/actions/runs/25979870162) (job 76366528649, commit `29eb26d` = A33) hit the GitHub Actions 6h0m0s max-execution-time limit and was force-cancelled (`CodeQL job status was configuration error`; orphan `actool` pid 70689 + `xcodebuild` pid 25560 terminated at cleanup). The next push's `swift, macos-26` run is the free natural re-exercise. **If it hangs again on an unrelated commit (chronic)**, escalate with one of: (1) wrap the `Build iOS app (swift)` step in `timeout <N>m` so a stuck `actool` is reaped and the job fails fast instead of burning 6h of macos-26 minutes; (2) add a workflow-level retry on the swift CodeQL job (mirrors the proposed XCUITest/Maestro retry above); (3) switch the swift CodeQL build to `-destination 'generic/platform=iOS Simulator'` (matches `make ios-build`; skips per-device asset thinning so `actool` runs the non-`--compress-pngs`/non-thinning path, sidestepping the hang surface) — verify CodeQL still gets a complete trace under generic-destination first.
