# ADR-006: CI/CD Code Signing and Distribution Strategy

## Status

Accepted

## Context

Knit Note targets both Android and iOS. The release workflow (`release.yml`) already produces a signed Android APK via base64-encoded keystore stored in GitHub Secrets. However, the iOS pipeline only builds the shared KMP framework — the iOS app is never archived, signed, or distributed.

To reach TestFlight (and eventually the App Store), the CI pipeline must:

1. Import an Apple Distribution certificate and provisioning profile into the CI runner.
2. Archive the iOS app with `xcodebuild archive`.
3. Export a signed IPA with `xcodebuild -exportArchive`.
4. Upload the IPA to App Store Connect for TestFlight distribution.

Key constraints:

- The project uses **GitHub Actions** directly, without Fastlane.
- The iOS project is generated by **XcodeGen** from `iosApp/project.yml`.
- The shared KMP framework must be built by Gradle before `xcodebuild` can archive.
- Apple Developer Program enrollment and valid certificates are prerequisites.
- The pipeline should **gracefully degrade** when signing secrets are not configured.

## Decision

### Signing Approach

Use **native xcodebuild with manual code signing**. No Fastlane or third-party signing tools.

- **Certificate**: Apple Distribution `.p12` stored as `IOS_DISTRIBUTION_CERT_BASE64` GitHub Secret.
- **Provisioning Profile**: App Store distribution `.mobileprovision` stored as `IOS_PROVISIONING_PROFILE_BASE64`.
- **Temporary Keychain**: Created per CI run at `$RUNNER_TEMP/app-signing.keychain-db` with a random password. Destroyed in an `if: always()` cleanup step.
- **ExportOptions.plist**: Generated at CI time (not committed) to keep the repository account-agnostic. The provisioning profile name is extracted from the installed `.mobileprovision` at runtime.

### TestFlight Upload

Use `xcodebuild -exportArchive` with **App Store Connect API key** authentication parameters (`-authenticationKeyPath`, `-authenticationKeyID`, `-authenticationKeyIssuerID`). When the ExportOptions.plist `destination` is set to `upload`, xcodebuild handles both IPA export and App Store Connect upload in a single step — no separate upload tool required.

This avoids `xcrun altool --upload-app`, which was deprecated in Xcode 14 and may be removed in future Xcode versions.

### Conditional Execution

All iOS signing and upload steps are gated by secret availability checks:

- **Signing secrets** (`IOS_DISTRIBUTION_CERT_BASE64`, `IOS_DISTRIBUTION_CERT_PASSWORD`, `IOS_PROVISIONING_PROFILE_BASE64`, `APPLE_TEAM_ID`): required for archive and IPA export.
- **TestFlight secrets** (`APP_STORE_CONNECT_API_KEY_ID`, `APP_STORE_CONNECT_ISSUER_ID`, `APP_STORE_CONNECT_API_KEY_BASE64`): required for upload only.

When secrets are absent, the job still builds the shared framework artifact and emits a warning annotation — no failure.

### Version Numbering

- `MARKETING_VERSION` (CFBundleShortVersionString): derived from the git tag (e.g., `v0.2.0` → `0.2.0`).
- `CURRENT_PROJECT_VERSION` (CFBundleVersion): set to `github.run_number` to guarantee a unique, auto-incrementing build number per TestFlight upload.

### Required GitHub Secrets

| Secret | Purpose |
|--------|---------|
| `IOS_DISTRIBUTION_CERT_BASE64` | Base64-encoded .p12 distribution certificate |
| `IOS_DISTRIBUTION_CERT_PASSWORD` | Password for the .p12 file |
| `IOS_PROVISIONING_PROFILE_BASE64` | Base64-encoded .mobileprovision |
| `APPLE_TEAM_ID` | Apple Developer Team ID (10-char) |
| `APP_STORE_CONNECT_API_KEY_ID` | App Store Connect API key ID |
| `APP_STORE_CONNECT_ISSUER_ID` | App Store Connect issuer ID |
| `APP_STORE_CONNECT_API_KEY_BASE64` | Base64-encoded .p8 API key |

### Android Signing (Unchanged)

Android signing was established in Phase 9 and remains unchanged:

- `KEYSTORE_BASE64`: base64-encoded `.jks` keystore, decoded to `$RUNNER_TEMP/keystore.jks`.
- `KEYSTORE_PASSWORD`, `KEY_ALIAS`, `KEY_PASSWORD`: signing credentials.
- Falls back to debug signing when secrets are absent.

## Consequences

### Positive

- No additional build tool dependencies (no Fastlane, no Ruby runtime).
- Secrets never persist on disk — temporary keychain and files are cleaned up after every run.
- Graceful degradation: contributors without Apple Developer accounts can still trigger releases (Android-only output).
- Reproducible: the same tag always produces the same version metadata.
- The signed IPA is attached to the GitHub Release alongside the Android APK.

### Negative

- More verbose workflow YAML compared to a Fastlane-based approach.
- ExportOptions.plist generation depends on provisioning profile introspection — fragile if the bundle ID changes.
- `xcodebuild -exportArchive` with `-authenticationKeyPath` requires the API key to be decoded to disk temporarily (cleaned up after each run).
- Provisioning profiles expire annually and must be renewed manually.

### Neutral

- Requires Apple Developer Program enrollment ($99/year) as a prerequisite for iOS distribution.
- TestFlight builds require App Store Connect review for external testing (internal testing is immediate).

## Alternatives Considered

| Alternative | Pros | Cons | Why Not Chosen |
|------------|------|------|----------------|
| Fastlane (match + pilot) | Rich ecosystem, `match` manages profiles automatically, `pilot` handles TestFlight | Adds Ruby dependency, another tool to maintain, learning curve | Unnecessary complexity for a single-app project |
| Xcode Cloud | Native Apple CI, automatic signing | Vendor lock-in, limited customization, separate from GitHub Actions | Project already uses GitHub Actions; splitting CI increases complexity |
| App Store Connect REST API (direct) | Future-proof, full control | More code to write (multipart upload, JWT auth), harder to debug | `xcodebuild -exportArchive` with auth key params is simpler and Apple-supported |
| Committed ExportOptions.plist | Simpler workflow, no runtime plist generation | Couples repo to specific Apple account, leaks team ID | Generated approach is more portable |
