# i18n Convention (Phase 33)

Last updated: 2026-05-02.

## Goals

1. Every user-visible string on Android and iOS resolves through a
   localization resource, not a hardcoded literal.
2. Both platforms share the **same key set**. A CI check fails the build
   if the sets drift.
3. Japanese and English are both first-class at launch. Additional
   locales are added by translating values only; keys do not change.

## Source of truth

| Locale | Shared Compose (`Res.string.*` / `Res.plurals.*`) | iOS SwiftUI |
|---|---|---|
| English | `shared/src/commonMain/composeResources/values/strings.xml` | `iosApp/iosApp/Resources/Localizable.xcstrings` (`"en"`) |
| Japanese | `shared/src/commonMain/composeResources/values-ja/strings.xml` | `iosApp/iosApp/Resources/Localizable.xcstrings` (`"ja"`) |

Three files total — Compose en, Compose ja, iOS String Catalog (which
holds both locales). All user-facing UI on both platforms resolves
through one of these three.

- **`shared/.../composeResources/values/`** — every shared Compose
  Multiplatform screen reads through here via
  `import io.github.b150005.skeinly.generated.resources.Res`
  + `stringResource(Res.string.<key>)` (or `pluralStringResource(Res.plurals.<key>, count, …)` for plural keys). Shared
  commonMain Kotlin cannot reference Android's `R`, so the Compose
  resources plugin is the only path.
- **`iosApp/.../Localizable.xcstrings`** — every SwiftUI screen in
  `iosApp/`. The iOS app does not render shared Compose screens — iOS
  is fully native SwiftUI from Phase 6 onward — so its strings are
  managed independently. `project.yml` declares
  `CFBundleDevelopmentRegion: en` and `CFBundleLocalizations: [en, ja]`.

`./gradlew verifyI18nKeys` enforces that all three files carry
identical key sets. Drift fails the build.

`androidApp/src/main/res/values/strings.xml` is **not** part of the
i18n parity set. It carries only Android-platform resources that the
manifest (or other native overlays) must reference via `@string/*` —
currently just `app_name`. User-facing copy never lives there; nothing
in `androidApp/` Kotlin reads `R.string.*` / `R.plurals.*`.

## Key naming

- `snake_case`.
- Prefix by role, not by screen:
  - `action_*` for buttons, menu items, accessibility labels that name an
    action (`action_save`, `action_undo`).
  - `state_*` for load / empty / idle state labels (`state_loading`).
  - `error_*` for error-message text (`error_load`, `error_save`).
  - `dialog_*_title` / `dialog_*_body` for confirmation dialogs that pair
    a title with a distinct body string. Single-button error alerts that
    do not have a paired body key (the alert body is the raw runtime
    error message) use `title_error` rather than `dialog_error_title` to
    avoid implying a non-existent `dialog_error_body` partner key.
  - `title_*` for screen titles that do not double as actions.
  - `body_*` for descriptive paragraph body copy paired with a title
    outside of a dialog or empty-state context (e.g. onboarding page
    body text next to a `title_*` headline). Use this prefix when the
    body is not tied to a `state_*` or `dialog_*` primary.
  - `label_*` for non-action labels (form field labels, status chips,
    dropdown option labels).
  - `hint_*` for placeholder/hint text in input fields (`hint_search_projects`).
  - `message_*` for transient feedback shown after a user action completes —
    Snackbar / Toast / Alert body text that is neither an error nor a
    persistent state label (`message_profile_updated`). Distinct from
    `state_*` (which covers load / empty / idle state labels, not
    action-completion feedback) and from `error_*` (which covers failure
    messages).
- Only add a screen prefix when the same word legitimately means two
  different things in two places (rare). Prefer role-prefixed reusable
  keys.

## How to reference

**Shared Compose Multiplatform (`commonMain`)** — every shared screen:

```kotlin
import io.github.b150005.skeinly.generated.resources.Res
import io.github.b150005.skeinly.generated.resources.action_save
import org.jetbrains.compose.resources.stringResource

Text(stringResource(Res.string.action_save))
```

The `Res` class is generated by the `compose.components.resources`
plugin (configured in `shared/build.gradle.kts` under the
`compose.resources { }` block). It reads from
`shared/src/commonMain/composeResources/values{,-ja}/strings.xml`.

**iOS (SwiftUI)**:

```swift
Text("action_save")   // Text(_:tableName:) auto-localizes
Button("action_save") { ... }
```

When you need a `String` (not a `Text`), use `String(localized:)`:

```swift
let label = String(localized: "action_save")
```

SwiftUI automatically resolves keys against `Localizable.xcstrings` when
the literal matches a key. Do not wrap with `NSLocalizedString` for
plain strings. (Plurals are the exception — see below.)

### Which SwiftUI constructors auto-localize a bare `String` literal?

| Auto-localizes (bare literal is fine) | Needs explicit `LocalizedStringKey(...)` wrap |
|---|---|
| `Text("key")` | `.alert("key", isPresented: ...)` — overloads for `String` and `LocalizedStringKey` exist; overload resolution on a bare literal can select the `String` form, so wrap explicitly. |
| `Button("key") { ... }` | `.navigationTitle("key")` for a computed/conditional expression (the `String`-typed overload may win). Use `.navigationTitle(LocalizedStringKey("key"))`. |
| `Label("key", systemImage: "...")` | Ternary / conditional keys passed to `Text`: use `Text(LocalizedStringKey(isA ? "key_a" : "key_b"))`. |
| `Section("key")` | Any API that takes `String` in one overload and `LocalizedStringKey` in another, where literal-promotion is ambiguous. When in doubt, wrap. |

Rule of thumb: if the constructor's primary label parameter is typed
`LocalizedStringKey` (e.g. `Text(_:)` via `ExpressibleByStringLiteral`),
a bare literal works. If the API has both `String` and
`LocalizedStringKey` overloads, wrap explicitly to avoid silent
literal-as-String resolution.

## Plurals

When a string contains a numeric quantity that drives singular / plural
agreement (English `1 row` vs `2 rows`, etc.), declare it as
`<plurals>` instead of `<string>`.

**Compose XML (both `values/` and `values-ja/`):**

```xml
<plurals name="label_rows_current">
    <item quantity="one">%1$d row</item>
    <item quantity="other">%1$d rows</item>
</plurals>
```

`values-ja` ships only the `other` variant — Japanese has no
grammatical plural distinction and CLDR maps every quantity to `other`.
Other locales follow their own CLDR rules (Russian needs `one` /
`few` / `many`, Polish needs `one` / `few` / `many` / `other`, etc.).

**Compose call site:**

```kotlin
import org.jetbrains.compose.resources.pluralStringResource
import io.github.b150005.skeinly.generated.resources.label_rows_current

Text(pluralStringResource(Res.plurals.label_rows_current, count, count))
```

The `count` argument appears twice — once to pick the plural variant,
once as the format substitution. When the format string has more than
one `%d`, pass the plural-driving count first and the rest in
positional order.

**iOS xcstrings — single-arg plurals:**

```json
"label_rows_current": {
  "extractionState": "manual",
  "localizations": {
    "en": {
      "variations": {
        "plural": {
          "one":   { "stringUnit": { "state": "translated", "value": "%lld row"  } },
          "other": { "stringUnit": { "state": "translated", "value": "%lld rows" } }
        }
      }
    },
    "ja": {
      "variations": {
        "plural": {
          "other": { "stringUnit": { "state": "translated", "value": "%lld 行" } }
        }
      }
    }
  }
}
```

Use `%lld` (Int64) — Swift `Int` is 64-bit on every supported target.

**iOS xcstrings — multi-arg plurals (plural rule applies to a
non-first arg):**

```json
"label_rows_of_total": {
  "localizations": {
    "en": {
      "stringUnit": { "state": "translated", "value": "%1$lld / %#@total@" },
      "substitutions": {
        "total": {
          "argNum": 2,
          "formatSpecifier": "lld",
          "variations": {
            "plural": {
              "one":   { "stringUnit": { "state": "translated", "value": "%lld row"  } },
              "other": { "stringUnit": { "state": "translated", "value": "%lld rows" } }
            }
          }
        }
      }
    },
    ...
  }
}
```

The outer `stringUnit.value` uses `%#@name@` for the substituted slot,
and the `substitutions[name].argNum` points at the arg that drives
plural selection.

**iOS Swift call site:**

```swift
Text(String.localizedStringWithFormat(
    NSLocalizedString("label_rows_current", comment: ""),
    Int(count)
))
```

Use `String.localizedStringWithFormat(...)` (locale-aware), **not**
`String(format: …)` — the latter is locale-naive and bypasses the
plural variation. Cast Int32 inputs from the Kotlin bridge to `Int`
so they line up with `%lld`.

When a single string spans multiple independent counts (e.g. `n added ·
m modified · k removed`), Android plurals only handle one quantity at
a time — split into separate plural keys (`label_diff_added`,
`label_diff_modified`, `label_diff_removed`) and concatenate at the
call site.

## Adding a key

1. Add the entry to **all three** sources:
   - `shared/src/commonMain/composeResources/values/strings.xml`
   - `shared/src/commonMain/composeResources/values-ja/strings.xml`
   - `iosApp/iosApp/Resources/Localizable.xcstrings`
2. Run locally:
   ```
   ./gradlew verifyI18nKeys
   ```
3. Key sync is enforced in CI — missing a platform will fail the build.

## Removing a key

Remove from all three sources in the same commit. The verifier will
catch partial removals.

## Non-goals in Phase 33

- Migration of existing screens. Existing hardcoded strings stay
  hardcoded until a focused i18n sweep is scheduled. Net-new UI (Phase
  32 editor onward) writes localized from day one.
- Symbol Catalog bilingual labels. `SymbolCatalog` currently exposes
  both JA and EN labels on every glyph (per the Phase 30.1 gallery
  design). A later phase will pick locale-appropriate labels via
  `Locale.current`; for now both remain visible.
